# Моделирование динамики социально-эмоциональных взаимодействий в коллективах дискретных автоматов

Данный проект представляет собой программный комплекс для исследования динамики социально-психологической структуры отношений в коллективах агентов. Проект реализован в рамках научно-исследовательской работы факультета Прикладной математики и информатики ТФ МГУ им. М.В. Ломоносова.

Коллектив агентов моделируется как сложная многоагентная система дискретных автоматов, состояние которых изменяется под влиянием эмоциональных факторов, индивидуальных поведенческих архетипов и реляционных связей.

## 1. Теоретическое описание модели

Коллектив описывается как кортеж $K = \{A_1, A_2, \dots, A_N\}$, управляемый синхронизирующим автоматом верхнего уровня. Каждый агент $A_i$ представляет собой дискретный автомат со следующим набором параметров:

1. **Вектор эмоционального состояния** $E_i(t) \in [-30, 30]^7 \cap \mathbb{Z}^7$:
   * Печаль — Радость
   * Страх — Спокойствие
   * Гнев — Смирение
   * Отвращение — Принятие
   * Привычка — Удивление
   * Стыд — Уверенность
   * Отчуждение — Открытость (Любовь)

2. **Реляционная матрица направленных связей** для каждой пары $(i, j)$ с целыми значениями на шкале $E_{21} = [-100, 100]$:
   * Выгода ($U_{ij}$) — полезность контакта;
   * Симпатия ($A_{ij}$) — эмоциональная предрасположенность;
   * Доверие ($T_{ij}$) — уровень надежности взаимодействия;
   * Чувствительность к частоте контактов ($R_{ij}$).

3. **Индивидуальный архетип** — неизменяемая во времени стратегия поведения, определяющая функции выбора партнёров, реакцию на отказы, динамику угасания и обновления отношений.
   * *Доступные в GUI архетипы*: Аналитик, Нонконформист, Медиатор, Конкурент, Оппортунист, Протектор, Деструктор, Инноватор, Рефлектор.

4. **Коэффициент чувствительности** $s_i \in (0, 3]$ — параметр, масштабирующий величину и инерционность изменения эмоциональных состояний и реляционных оценок.

### Цикл взаимодействия
На каждом временном шаге (такте симуляции) агенты классифицируют окружение на приоритетных, опциональных и избегаемых партнеров, осуществляют выбор кандидата по правилу Softmax и совершают парный акт общения $\sigma_{ij} \in \{-1, 0, 1\}$ (успех, отказ, неудача), на основании которого пересчитываются эмоциональные векторы и оценки отношений.

## 2. Архитектура и технические особенности

Программный комплекс объединяет высокоуровневое управление на Python и оптимизированный параллельный вычислительный блок на C++:

1. **Параллельное C++ ядро (Engine)**:
   * Реализовано на языке C++ для скоростной обработки $O(N^2)$ взаимодействий;
   * Использует многопоточность **OpenMP** для параллелизации расчётов ежедневного цикла состояний;
   * Связывается с Python-оболочкой через скомпилированные биндинги `emotion_engine`.

2. **Интеграция ClickHouse**:
   * Обеспечивает асинхронное сохранение детальных логов симуляции по дням и слотам (Big Data);
   * Облегчает сбор агрегированной статистики для последующего анализа динамики устойчивости отношений.

3. **Графический интерфейс (GUI)**:
   * Отображает граф взаимосвязей в реальном времени;
   * Содержит детальный Мастер инициализации Университета с автоматическим распределением по факультетам, потокам и группам (1875+ агентов) с расписанием и ограничением вместимости.

## 3. Установка и запуск

### Системные требования
* Python версии 3.10 или новее;
* Компилятор C++ с поддержкой стандарта C++17 и OpenMP (например, `clang`/`gcc`);
* СУБД ClickHouse (опционально, для асинхронного Big Data логирования).

### 1. Автоматическая подготовка окружения
Для создания виртуального окружения и установки необходимых зависимостей:

**Для macOS / Linux:**
```bash
chmod +x install.sh
./install.sh
```

**Для Windows (запустить из консоли `cmd`):**
```cmd
install.bat
```

### 2. Сборка вычислительного ядра C++
При модификации C++ исходников в папке `core/src/` перекомпиляция осуществляется скриптом:

**Для macOS / Linux:**
```bash
chmod +x build.sh
./build.sh
```

**Для Windows (требуется компилятор `g++` / MinGW в PATH):**
```cmd
build.bat
```

### 3. Запуск симулятора
Для старта симулятора с интерактивной графической панелью:

**Для macOS / Linux:**
```bash
chmod +x run.sh
./run.sh
```

**Для Windows:**
```cmd
run.bat
```

Вы также можете запустить систему в фоновом (headless) режиме исследований напрямую через Python:
```bash
# macOS / Linux
./venv/bin/python main.py --scenario scenarios_archive/Run03_Harmony_MIXED.json --silent --steps 100

# Windows
venv\Scripts\python main.py --scenario scenarios_archive\Run03_Harmony_MIXED.json --silent --steps 100
```

## 4. Развертывание Telegram-бота мониторинга

Для удалённого наблюдения за длительными фоновыми сериями симуляций (на 4+ виртуальных года) в системе предусмотрен Telegram-бот мониторинга (`scripts/tg_log_watcher.py`). Бот позволяет отслеживать активность сервера, потребление ресурсов и прогресс записи в СУБД ClickHouse.

### Настройка конфигурации (.env)
Перед запуском бота убедитесь, что в файле `.env` прописаны соответствующие параметры:
```env
TELEGRAM_BOT_TOKEN="ваш_токен_бота"
TELEGRAM_CHAT_ID="идентификатор_вашего_чата"
```

### Запуск в фоновом режиме (демоне)
Чтобы бот работал непрерывно в фоне после закрытия терминала:

**Для macOS / Linux:**
```bash
nohup ./venv/bin/python scripts/tg_log_watcher.py > data/output/tg_bot.log 2>&1 &
```

**Для Windows (через фоновую задачу cmd):**
```cmd
start /B venv\Scripts\python scripts\tg_log_watcher.py > data\output\tg_bot.log 2>&1
```

### Функционал бота (кнопки быстрого управления)
При отправке команды `/start` или `/help` бот выведет меню со следующими кнопками:
* 📊 **Статус** — вывод текущего состояния симуляции (активен ли процесс, его PID, а также последние 10 строк из лога `simulation_batch.log`).
* 📈 **Прогресс** — запрос текущего шага и игрового дня в базе данных ClickHouse для каждого активного сценария с расчётом виртуального месяца симуляции.
* 🖥 **Ресурсы** — подробная сводка о загрузке CPU, распределении по ядрам, использовании оперативной памяти (RAM), файла подкачки (Swap) и свободного места на жестком диске.
* 📜 **Фулл логи** — вывод последних 3000 символов из общего текстового файла лога.
* 🚀 **Рестарт** — безопасное принудительное завершение всех фоновых процессов симулятора, очистка таблиц ClickHouse (`TRUNCATE`) и перезапуск симуляции с чистого листа.

## 5. Развертывание и настройка СУБД ClickHouse

СУБД **ClickHouse** используется в качестве аналитического хранилища для накопления и постобработки терабайт логов (состояния агентов, ежедневные матрицы отношений, пошаговые события) при долгосрочном моделировании.

### 1. Быстрый запуск локального сервера (Docker)
Рекомендуется использовать официальный образ ClickHouse в контейнере Docker:

```bash
docker run -d \
  --name clickhouse-server \
  -p 8123:8123 \
  -p 9000:9000 \
  --ulimit nofile=262144:262144 \
  -e CLICKHOUSE_DB=default \
  -e CLICKHOUSE_USER=default \
  -e CLICKHOUSE_PASSWORD=clickhouse_pass \
  clickhouse/clickhouse-server
```

### 2. Инициализация таблиц базы данных
Для создания необходимых структур хранения примените SQL-скрипт схемы `data/clickhouse_schema.sql` удобным способом:

* **Способ А (прямым HTTP-запросом):**
  ```bash
  curl -u default:clickhouse_pass -d @data/clickhouse_schema.sql http://localhost:8123/
  ```

* **Способ Б (через консоль внутри контейнера):**
  ```bash
  docker exec -i clickhouse-server clickhouse-client --user default --password clickhouse_pass --queries-file - < data/clickhouse_schema.sql
  ```

### 3. Настройка параметров подключения (.env)
Параметры соединения с СУБД можно переопределить в файле `.env` в корне проекта:
```env
CLICKHOUSE_HOST="localhost"
CLICKHOUSE_PORT=8123
CLICKHOUSE_USER="default"
CLICKHOUSE_PASSWORD="clickhouse_pass"
CLICKHOUSE_SECURE="False"
```

Если база данных недоступна по указанным реквизитам, регистратор выведет предупреждение в консоль и продолжит работу симуляции в штатном режиме, отключив запись в БД.
